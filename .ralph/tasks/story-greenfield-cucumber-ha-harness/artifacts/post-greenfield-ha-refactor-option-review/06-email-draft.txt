I finished the intentionally weird stop between greenfield test creation and bug fixing. The short version is that the greenfield HA suite and the preserved bug evidence are all pointing at the same larger problem: authority is still being decided by several partially different models. Quorum in `src/dcs/state.rs` still uses the observed-member shortcut instead of configured-majority rules, `src/ha/decide.rs` still collapses every non-`FullQuorum` trust state into fail-safe behavior much too bluntly, `src/ha/worker.rs` still relies on a local dedup shortcut that bypasses the authoritative effect path, and `src/runtime/node.rs` still chooses startup intent from a looser rule set than the steady-state HA loop uses.

I wrote three coherent refactor directions under `.ralph/tasks/story-greenfield-cucumber-ha-harness/artifacts/post-greenfield-ha-refactor-option-review/`. The best option is `03-option-unified-authority-snapshot.md`, which keeps the current `ha_loop` and pure `decide()` style but introduces one explicit authority snapshot and one deterministic durability-aware ranker shared by quorum evaluation, leader election, switchover validation, startup intent, recovery, and rejoin visibility. That is the best balance because it fixes the whole failure inventory as one model without forcing a total rewrite. The second-best option is `02-option-authority-layer-convergence.md`, which is more conservative and probably easier to stage, but it leaves more of the current shape intact and therefore leaves more room for the same rule drift to return later.

The comparison matrix is in `05-comparison-matrix.md` and the evidence inventory is in `01-evidence-inventory.md`. One important truth boundary is that the repo is greener now than when this stop was conceived, so the inventory separates current gate results from preserved bug-task evidence captured on March 10 and March 11, 2026. I did not pretend those historical failures are still reproducing if the current run is green, but I also did not ignore them, because they are the most concrete proof of what design pressure the greenfield suite exposed.

The architectural questions I think matter most before implementation starts are these. Do you want startup authority to be fully unable to infer primary intent without the same authority snapshot the steady-state loop would trust, even if that makes bootstrap and resume more conservative? Do you want durability safety to win over availability when no candidate can prove it is on the best known timeline and WAL position, or do you want a bounded degraded mode for certain two-node cases? Should explicit fail-safe remain a visible phase only for hard stop conditions, with other degraded states represented separately, or do you prefer keeping the external surface smaller even if the internal model becomes richer? And if Option B lands cleanly, do you want to stop there, or treat it as the bridge toward the larger single authority machine in `04-option-single-ha-authority-machine.md`?
