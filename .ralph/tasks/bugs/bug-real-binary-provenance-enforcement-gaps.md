## Bug: Real-binary provenance enforcement gaps in installers and harness <status>done</status> <passes>true</passes> <passing>true</passing>

<description>
Real-binary tooling currently enforces existence/executability but not strong provenance at runtime.

Detected during skeptical audit of `tools/install-postgres16.sh`, `tools/install-etcd.sh`, and `src/test_harness/binaries.rs`.

Key gaps:
- `src/test_harness/binaries.rs` validates only `exists + regular file + executable`; it does not verify ownership, canonical target policy, version, or checksum.
- Symlinked binaries are accepted implicitly via `fs::metadata`, so `.tools/*` can point outside expected roots without rejection.
- `tools/install-postgres16.sh` trusts package-manager state + version string for major `16.x`, but does not pin exact package provenance in-repo and links `.tools/postgres16/bin/*` to mutable system paths.
- Installer scripts invoke privileged/sensitive commands by name (`sudo`, `dnf`, `curl`, etc.) via PATH resolution and do not harden PATH trust assumptions.

Please explore and research the full codebase first, then implement fail-closed provenance verification for real-binary prerequisites and installer trust boundaries.
</description>

## Plan

### Goals and trust model (fail-closed)

- Treat **repository-local `.tools/` binaries as required test prerequisites**, but do not implicitly trust them.
- Fail closed if:
  - a required binary is missing,
  - the path is a symlink (or resolves outside `.tools/`),
  - the binary is not a regular executable file,
  - the binary does not match a checksum/version rule generated by the installer,
  - installer trust assumptions are violated (unsafe PATH / untrusted command resolution).

### Phase 1 — Provenance data model: repo-tracked policy + local attestation (fail-closed)

- [x] Define a small, stable *policy* format that is committed to git:
  - Location: `tools/real-binaries-policy.json` (tracked).
  - Purpose: external, reviewable source of truth for **which binaries are required** and what versions are acceptable.
  - Contents (per binary entry):
    - `label` (e.g. `postgres`, `pg_ctl`, `pg_rewind`, `initdb`, `pg_basebackup`, `psql`, `etcd`)
    - `path` (repo-relative path under `.tools/…`)
    - `expected_version` (structured, so installers don’t need to embed arbitrary version strings in JSON):
      - etcd: `{ "kind": "etcd", "exact": "3.6.8" }`
      - postgres tools: `{ "kind": "postgres", "major": 16 }` (optionally later extended to pin a patch line)
    - for download-based tools (etcd): include the pinned archive sha256 per-arch in policy.
    - for system-provided tools (Postgres on AlmaLinux/RHEL): include `allowed_target_prefixes` so `.tools/...` may be a symlink, but only to allowlisted system directories.
- [x] Define a local *attestation* manifest written by installers:
  - Location: `.tools/real-binaries-attestation.json` (ignored by git, but required locally).
  - Purpose: machine-specific tamper detection for **the actual installed/copy-materialized files**.
  - Contents (per binary entry):
    - `label`
    - `path` (must match policy)
    - `sha256` (hex)
    - `size_bytes` (integer; cheap extra tamper signal)
    - `installed_at_utc` (debug only)
  - Manifest-level fields:
    - `schema_version` integer
    - `generated_by` (installer script name + version)
- [x] Ensure policy+attestation are **complete**: missing required entry is a hard error.

Rationale:
- Policy is reviewable and stable across machines; attestation is local and detects post-install substitution.
- This avoids a pure “trust on first use” model for etcd (archive checksum is pinned in policy), while keeping Postgres grounded in the OS package trust model plus local tamper detection.

### Phase 2 — Harden installer scripts (trust boundary + deterministic output)

#### 2A. Common script hardening helper

- [x] Add `tools/install-common.sh` and source it from installers.
- [x] In `tools/install-common.sh`:
  - Set a conservative PATH early (include `/usr/local/*` to avoid brittle failures on valid systems):
    - default: `PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin`
  - Provide `require_cmd_path <name>` that resolves an absolute path once and enforces:
    - path is absolute
    - path is a regular file
    - executable bit set
    - if the command path is a symlink, validate the resolved target is also allowlisted (but execute via the symlink path to preserve argv[0] semantics for multi-call shims like rustup’s `cargo`)
    - (optional strict mode) path prefix is in an allowlist; allowlist override via env for debugging, but fail-closed by default with a clear error message
  - Provide small helpers for:
    - `sha256_file <path>` (using `sha256sum` with strict parsing)
    - `file_size_bytes <path>`
    - safe temp file/dir creation (`mktemp`)
    - lock-protected, atomic manifest update:
      - take an exclusive lock (e.g. `flock`) on a `.lock` file next to `.tools/real-binaries-attestation.json`
      - read+merge existing JSON (if present) instead of clobbering
      - write to temp then `mv` atomically

#### 2B. etcd installer (`tools/install-etcd.sh`)

- [x] Switch command invocation to absolute paths from `require_cmd` (no bare `curl`, `tar`, `sha256sum`, etc.).
- [x] Tighten download behavior:
  - Use strict curl flags (`--fail --location --show-error`, enforce HTTPS, add retries).
  - Use `mktemp` for temporary archive path (avoid predictable `*.tmp.$$`).
- [x] After install, compute sha256 for the installed `.tools/etcd/bin/etcd` and append/update the manifest entry:
  - Verify pinned archive sha256 as today.
  - Verify `etcd --version` output matches the pinned version.
  - Update `.tools/real-binaries-attestation.json` by merge (do not overwrite Postgres entries).

#### 2C. Postgres installer (`tools/install-postgres16.sh`)

- [x] Keep `.tools/postgres16/bin/*` as symlinks to system binaries, but **fail-closed and explicit**:
  - Verify each `.tools/postgres16/bin/*` symlink resolves only into allowlisted trusted prefixes.
  - Record `resolved_path_abs` in the attestation manifest so retargeting is detected.
  - Continue to require Postgres major `16` via `--version`.
- [x] Replace `source /etc/os-release` with a non-executing parser (avoid executing file contents).
- [x] Tighten trust assumptions:
  - Resolve `sudo`, `dnf`, `cp`, `chmod`, `mv`, `mkdir`, `readlink` via `require_cmd_path` and use resolved absolute paths for invocation.
  - Reject if helpers resolve to symlinks or non-executables; keep allowlist strict-by-default but overridable with an explicit env var for debugging.
- [x] Pin and verify version policy:
  - Continue to require Postgres major `16` for all copied tools via `--version`.
  - (Optional but recommended) tighten to `16.<minor>` pin once we decide an exact patch line.
- [x] (Optional but recommended on RPM systems) add an OS package integrity check step:
  - verify package presence (e.g. `rpm -q postgresql postgresql-server`) and run `rpm -V` for those packages; fail closed if verification reports unexpected changes.
- [x] Generate/refresh manifest entries for each installed Postgres tool:
  - hash each allowlisted resolved binary under `.tools/postgres16/bin/*`
  - record `size_bytes`
  - update `.tools/real-binaries-attestation.json` by merge (do not overwrite etcd entries)

### Phase 3 — Harness-side fail-closed verification used by real-binary tests

- [x] Add a Rust verifier module (suggested: `src/test_harness/provenance.rs`) that:
  - loads `tools/real-binaries-policy.json` and `.tools/real-binaries-attestation.json`,
  - validates required entries exist and match (policy ↔ attestation),
  - verifies each required binary path is:
    - under `.tools/` as a repo-relative entry path,
    - no symlink in any ancestor component under `.tools/` (reject symlinked directories),
    - leaf symlink allowed only when policy provides an allowlist; canonical target must be allowlisted,
    - a regular file with executable bit on Unix,
    - sha256 matches the manifest entry,
    - version matches policy (after sha verification; do not execute unverified binaries),
    - basic safety invariants (Unix only):
      - file is not group/other writable
      - manifest file is not group/other writable
  - caches successful verification results within the process (e.g. `OnceLock`) to avoid repeated hashing/version probes inside the 120s `make test` wall-clock.
- [x] Refactor `src/test_harness/binaries.rs` so that:
  - `require_etcd_bin_for_real_tests()` and `require_pg16_process_binaries_for_real_tests()` call the provenance verifier (not just `exists + executable`).
  - Prefer implementing a parameterized internal helper like `real_binaries_from_repo_root(repo_root: &Path) -> …` so unit tests can run against a temp “fake repo” without touching the real `.tools/`.
- [x] Do **not** introduce `unwrap/expect/panic` in the verifier; all failures must return `HarnessError::InvalidInput(...)` with actionable detail.

### Phase 4 — Deterministic negative-control tests (tampering fails early)

- [x] Add unit tests (in `src/test_harness/provenance.rs` or `src/test_harness/binaries.rs`) that:
  - Create a temp directory shaped like a repo root, with:
    - `.tools/postgres16/bin/<required tools>`
    - `.tools/etcd/bin/etcd`
    - `tools/real-binaries-policy.json`
    - `.tools/real-binaries-attestation.json`
  - Use tiny, deterministic “fake executables” (e.g. shell scripts with `--version` output) so the version-check path is exercised without relying on host binaries.
  - Then prove fail-closed behavior:
    - checksum mismatch: modify one fake binary after writing the manifest → verification must fail
    - symlink rejection: replace a binary with a symlink to another file → verification must fail
    - path escape: make `.tools/postgres16/bin/postgres` be a symlink (or directory symlink) that resolves outside `.tools` → verification must fail

This satisfies “substituted `.tools` binaries fail early and loudly” without relying on live system binaries.
